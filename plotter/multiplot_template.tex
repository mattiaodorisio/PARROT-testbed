The following figures shows the performance of DeLI index under different configurations. The benchmark has been executued as follows: for each dataset, we run three workloads: existing lookup, in-distribution lookup, in-distribution insertion, deletions, and mixed insertions and lookups on a batch of $k = 8192$ operations \note{there is no particular reason for this value, we can discuss a better choice}. The plots below show the total time to execute the batch of $k$ operations.
\paragraph*{Workloads}
\begin{itemize}
  \item Existing Lookup: searching for $k$ keys already present within the dataset.
  \item In-distribution Lookup: searching for $k$ keys that do not belong to the same dataset, but whose distribution follows the same of the dataset. Obtained by unifromly sampling one of n+1 intervals between keys and then uniformly within that interval
  \item In-distribution Insertion: inserting $k$ keys drawn from the same distribution of the dataset.
  \item Deletion: deleting $k$ keys already present within the dataset.
  \item Mixed: a mix of 50\% lookups and 50\% insertions, all drawn from the same distribution of the dataset.
\end{itemize}

The following plots compare the many possible configuartions of the DeLI index. Each configuration is defined by the following parameters:
\begin{itemize}
  \item Bottom level optimization. Possible values: N (None), SI (slot index: bitvector to jump to predecessor/successor), GFP (gap fill predecessor: the gaps store the predecessor), GFS (gap fill successor), GFB (gap fill both).
  \item Simd lanes. Number of simd batches used (Possible values: 0 for no simd, 1, 2, and 4).
  \item RHT load factor (\%).
  \item Top level optimization: Possible values: P (Precompute: precompute predecessor and successor of the RHTs, requires Static DeLI), BI (bucket index: bitvectors to search for the next/prev non-empty RHT table).
  \item Number of high bits: we discussed Stefan H to take all the bits in [1, 20]. \note{Due to the huge time and memory to compile the exploding combination of template parameters, for the moment I just benchmarked 4, 8, 12, 16, and 20 bits.}
\end{itemize}
Some of the optimizations are not possible in the dynamic DeLI. The last parameter of the DeLI configuration (512 in all figures) is the width of the available simd lanes (independently of the fact that simd is actually used or not.)

\section{Comparisons among indexes}
The plots below compare the performance of different indexes on various datasets by varying the initial dataset size. Please note, that the dataset size can change due to updates. 
All the workloads are run as a batch of $k$ operations. Each batch is repeated multiple times to obtain median of the execution time.
The workloads are the same of Section \ref{sec:deliconfig}.
\paragraph*{Competitors} (see the overleaf for the refs and descriptions)
\begin{itemize}
  \item ALEX
  \item LIPP \note{I got some failing assert in the LIPP library}
  \item RadixSpline
  \item PGM-index (static and dynamic version)
  \item TLX-Btree (Bingmann's implementation of standard B-tree)
\end{itemize}

\paragraph*{Notes}
\begin{itemize}
  \item Dynamic learned index ALEX, LIPP and PGM-dynamic are designed to support key-value mapping.
  \item LIPP library doesn't support search of non-existing keys. I would say its whole purpose is pointless as a hash table would be faster.
  \item ALEX and PGM support predecessor and successor, however they do not have a simple interface to store keys only.
  \item keys-only competitors are static.
  \item All competitors allow for duplicates, we are not.
\end{itemize}

\paragraph*{Datasets}
\begin{itemize}
\item 32 bit integer.
\item According to the distribution: scaled and rounded to the closer integer in $[2^{32}]$
\item Without duplicates: reject duplicates (is this ok? The density is the same?)
\end{itemize}
